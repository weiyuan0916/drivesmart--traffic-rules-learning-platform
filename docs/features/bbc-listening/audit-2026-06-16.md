# BBC Listening — Content Policy Audit & Migration

**Date:** 2026-06-16
**Auditor:** DriveSmart engineering (with help from Cursor assistant)
**Scope:** All BBC-related code under `backend/`, `src/features/listening/pages/bbc/`, `src/features/listening/api/`, `src/features/listening/stores/`, and supporting migrations / tests.
**Reference rule:** `.cursor/rules/bbc-feature.mdc`

---

## 1. Executive Summary

A 2026-06-16 audit discovered that the BBC feature, as implemented, violated the
content policy defined in `.cursor/rules/bbc-feature.mdc`. The most significant
violations:

- The `CrawlBbc6MinLessons` artisan command downloaded BBC transcript PDFs
  from `downloads.bbc.co.uk` and stored the parsed text in
  `listening_external_lessons.metadata_json.segments[]`.
- The `BbcTranscriptParser` service was used to split that text into
  ~3-5 second segments and feed it into the dictation feature, making
  DriveSmart effectively a rehost of BBC content.
- The `BbcService::scoreSegment` method used BBC transcript text as the
  ground truth for scoring user input.

This was contrary to the "companion learning platform" intent documented in the
rule. A user with the `BbcSeeder` populated would never see the issue because
the seeded data was hand-written, not crawled — but production crawls would
have downloaded and rehosted BBC content.

A migration to a "user_provided" dictation model has been completed. The
audit's findings and the migration plan are recorded here.

## 3. Migration Plan (Soft Audit)

The product team chose the "soft audit" path: **stop the violating
behavior, keep existing data, do not disrupt users who already have
dictation attempts stored**. The user experience for legacy rows changes
(text is no longer shown to the user), but progress data is preserved.

### 3.1 Phases

| Phase | Action | Risk |
|-------|--------|------|
| 1 | Disable `CrawlBbc6MinLessons` (no-op) and mark `BbcTranscriptParser` deprecated | Low |
| 2 | Add `segments_source` column to `listening_external_lessons` | Low |
| 3 | Split `BbcService` into `BbcCatalogService` + `BbcDictationService` | Low (facade preserves API) |
| 4 | Add BBC vocabulary metadata cache (`bbc_lesson_vocabulary_cache` table) | Low |
| 5 | Update tests to assert the new safety check | Low |
| 6 | Update documentation | None |

A subsequent "hard purge" phase is recommended but is **not** part of
this audit. It would delete `metadata_json.segments[]` from all
`listening_external_lessons` rows and drop the `UserExternalLessonSegment`
table (or repurpose it for the user-provided model only). This requires
product sign-off and a deprecation period for users.

### 3.2 What changed for users

| Scenario | Before | After |
|----------|--------|-------|
| User opens a BBC lesson with old `legacy_bbc` segments | Sees BBC transcript, types into it, gets scored against BBC text | Sees "BBC content retired" notice, prompted to supply own audio + transcript |
| User adds vocabulary to a BBC lesson | Saves to their notebook | Same — no change |
| User takes notes on a BBC lesson | Saves to their notebook | Same — no change |
| User uploads a DailyDictation lesson (Python crawler) | Works | Works — different system, unaffected |
| User runs `php artisan crawl:bbc-lessons` | Crawls metadata only | Same — no change |
| User runs `php artisan crawl:bbc-6min` | Downloaded BBC PDFs | Now prints deprecation notice |

## 4. New Service Architecture

The single `BbcService` class was split into two services. This makes the
content-policy boundary explicit at the call site and mirrors the
DailyDictation Python service, which is already split into
`topic_crawler` / `section_crawler` / `lesson_crawler` / `audio_slicer`.

```
BbcCatalogService (formerly BbcService.getSource/listLessons/getLesson/...)
  ├─ listing
  ├─ source management
  ├─ notes CRUD
  ├─ vocabulary CRUD
  ├─ progress tracking
  └─ dashboard metrics

BbcDictationService (formerly BbcService.getDictation/scoreSegment/...)
  ├─ session retrieval
  ├─ segment text retrieval (only for permitted sources)
  ├─ scoring (with content-policy gate)
  ├─ summary aggregation
  └─ completion

BbcVocabularyCacheService (new)
  ├─ vocabulary cache CRUD
  └─ bulk sync from crawler

DictationScoringService (shared, pure logic)
  └─ word-level scoring (independent of source)
```

`BbcService` remains as a thin facade that delegates to the three services
above. New code should depend on the split services directly.

## 5. New `segments_source` Column

```sql
ALTER TABLE listening_external_lessons
  ADD COLUMN segments_source VARCHAR(32) NULL,
  ADD CONSTRAINT listening_external_lessons_segments_source_check
    CHECK (segments_source IS NULL
        OR segments_source IN ('legacy_bbc', 'user_provided', 'curated', 'manual'));
```

| Value | Meaning | API text redaction | Scoring permitted |
|-------|---------|--------------------|-------------------|
| `NULL` (default) | Unknown / pre-migration | Yes | No |
| `legacy_bbc` | Crawled by deprecated `CrawlBbc6MinLessons` | Yes | No |
| `user_provided` | User typed it in their browser | No | Yes |
| `curated` | Hand-edited BBC content | Yes | No |
| `manual` | Hand-typed by an admin or teacher | No | Yes |

The default is "safe by default": unknown rows have their text stripped at
the API boundary, and `BbcDictationService` refuses to score against them.

## 6. Frontend Fallback UI

A new component `BbcDictationEmptyState` is shown when:
- `has_segments === false`, OR
- `segments_source === 'legacy_bbc'`, OR
- `requires_user_transcript === true` (set by the API for legacy rows)

The component guides the user to:
1. Open the original BBC lesson (link out, attribution preserved)
2. Upload their own audio file (kept in the browser only, never uploaded)
3. Paste their own transcript (split client-side into ~8-word segments)
4. Run the dictation loop entirely in the browser

Scoring is performed client-side by `DictationScoring` (mirror of
`DictationScoringService` in PHP).

## 7. Vocabulary Cache

A new table `bbc_lesson_vocabulary_cache` stores only:
- `word` (varchar 100)
- `brief_meaning` (varchar 500, one-line snippet)
- `position` (display order)

The cache is populated by `php artisan crawl:bbc-vocabulary`, which
scrapes the BBC episode page's "Vocabulary" section and stores only
the word + first paragraph of the visible definition. The brief_meaning
column is capped at 500 characters to ensure we never store BBC's full
glossary definitions.

API endpoints:
- `GET /api/v1/listening/bbc/{id}/vocabulary-cache`
- `POST /api/v1/listening/bbc/{id}/vocabulary-cache/sync`

The cache is meant to **suggest** vocabulary to the user; the user
still creates their own `UserExternalLessonVocabulary` entries, which
remain under their control.

## 9. Recommended Next Steps (Out of Scope)

1. **Hard purge (after user notification period):** delete
   `metadata_json.segments[]` from all rows, drop
   `UserExternalLessonSegment` table, remove `BbcSeeder` references to
   fake transcripts.
2. **Remove deprecated code:** delete `BbcTranscriptParser` and
   `CrawlBbc6MinLessons` once all tests are migrated (estimated 1-2
   sprints).
3. **Add BBC content policy banner to the BBC lesson detail page**
   explaining the "audio + transcript by you" model.
4. **Telemetry:** add a counter for `scoreSegment` returning null due to
   `legacy_bbc`, to know how many users are still on legacy rows.
